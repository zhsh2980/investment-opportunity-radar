# ADR 0001：钉钉为唯一触达渠道，Web 降级为管理后台

日期：2026-07-08
状态：已接受

## 背景

系统最初按「机会处理工作台」设计 Web 端：今日工作台（卡片流、倒计时、执行/观望/跳过三操作）、跟踪台（三列看板、金额/收益复盘、命中率统计）、历史搜索、系统设置四页，钉钉推送作为引流入口指向 Web 详情页。

上线后与用户核对真实使用流程，发现是：**钉钉收到提醒 → 在钉钉内看结论 → 直接去公众号看原文 → 自行决策**。不存在「回到系统点操作」「事后录入收益」的环节；Web 详情页作为中转反而多了一跳。

## 决策

1. 钉钉消息升级为自包含的**简报**：3-5 条 AI 要点 + 仅公众号原文链接（不再链接系统页面）。时效性以「要点首条写明具体日期」表达，Prompt 注入当前日期；不维护结构化 time_window 字段。
2. 砍掉工作台（含三操作按钮、倒计时、临期统计）、跟踪台、日报页；`opportunity_track` 表保留不删。历史搜索升级为首页，系统设置保留，分析详情页恢复登录保护（仅作后台下钻调试用）。
3. AI 输出扁平化为 6 字段（score / has_opportunity / summary / key_points / opportunity_types / no_opportunity_reason），停止写入 `Opportunity` 子表（历史数据保留）。
4. 信息源引入**机会类/宽泛类**分类，只影响推送策略：机会类命中阈值即推；宽泛类只进日报，≥80 分破例推送。取消「每天最多推 4 条」全局上限，推送量由源分类与阈值控制。

## 后果

- 正面：交付物与真实使用路径一致；Prompt 更短更稳，消除了 Prompt 与解析器的字段漂移问题；Web 维护面缩小一半以上；`/analysis` 无需公开访问，消除枚举泄露风险。
- 负面：放弃了结构化机会实体，未来若做机会去重或日历源核验（本轮明确不做，观察一个月再评估），需要重新引入结构化输出；复盘/命中率闭环被移除，无法用数据评估信息源质量，只能靠主观体感。
- 回退成本：页面代码可从 git 历史恢复，但 `Opportunity` 表将出现数据断层。

## 备选方案

- 保留工作台并培养回系统记录的习惯：与用户明确的使用意愿相悖，死功能持续产生维护成本。
- 只砍跟踪台、保留极简工作台：工作台失去三操作后只剩一个列表，与历史页重复。
